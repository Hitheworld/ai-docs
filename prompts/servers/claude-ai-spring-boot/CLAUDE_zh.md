### 1. 计划模式默认
- 对于任何非简单的任务（3 个以上步骤或架构决策），请进入计划模式。
- 验证步骤应使用计划模式，而不仅仅是构建步骤。
- 事先编写详细的规格说明，以减少歧义。

### 2. 自我提升循环
- 用户进行任何更正后：使用以下模式更新 `tasks/lessons.md`
  为自己制定规则，以避免犯同样的错误。
  反复实践这些经验教训，直到错误率下降为止。
- 在项目课程开始时回顾课程内容

### 3. 完成前的核实
- 永远不要在未证明任务有效的情况下将其标记为已完成。
- 主程序和你的修改之间在相关情况下的行为差异
  问问自己：“一名工程师会批准这样做吗？”
  运行测试，检查日志，验证正确性

### 4. 追求优雅（平衡）
- 对于非微不足道的更改：停下来问问自己“有没有更优雅的方法？”
- 如果感觉某个解决方案不够优雅：“以我现在所掌握的所有信息，实现一个更优雅的解决方案。”
- 如果是简单显而易见的解决方法，请跳过此步骤。不要过度设计。
  在展示作品之前，先审视一下自己的作品。

核心原则
- **简洁至上**：力求每个改动都尽可能简单，对代码的影响最小。
- **绝不偷懒**：找出根本原因，不搞临时修补。遵循高级开发人员的标准。

## 项目通用说明

- 始终使用依赖项的最新版本。
- 始终以 Spring Boot 应用程序的形式编写 Java 代码。
- 始终使用 Maven 进行依赖管理。
- 务必为生成的代码创建正反两方面的测试用例。
- 始终在 .circleci 目录中生成 CircleCI 流水线以验证代码。
- 尽量减少生成的代码量。
- Maven 构件名称必须与父目录名称相同。
- 对 Maven 项目使用语义化版本控制。每次生成新版本时，都要增加版本号中的 PATCH 部分。
- 使用 `pl.piomin.services` 作为 Maven 项目和基础 Java 包的组 ID。
- 请勿使用 Lombok 库。
- 生成 Docker Compose 文件以运行应用程序使用的所有组件。
- 每次生成新版本时，请更新 README.md。
